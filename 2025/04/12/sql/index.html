在MYSQL5.0版本后,系统会默认在数据库中存放一个informa_schema的数据库

手工技巧

这里主要想讲

如何采用手工的方式去发现sql注入漏洞

第一种

看页面是否异常 来判断

第二种

newspaper.php?Id=100

你输入

newspaper.php?Id=100-1

观察是否会返回99号新闻的内容。这个测试主要是判断后端有没有过滤、转义特殊字符，如果没有，大概率是能注的

变种还有别的数学运算符，比如^

newspaper.php?Id=100^1 正常返回页面

newspaper.php?Id=100^0 不正常

第三种

看回显的内容顺序上是否有些不同

第四种

就是字符型的判断方法

加单引号 双引号 括号 等等

第五种也很有意思，xiazhu插件里学到的，大概是咋样的呢，比方说

test.php?id=11’ //页面异常
test.php?id=11’’ //页面正常

这种手法也可以很好的判断注入点，原理也很简单，不管后端sql语句写成啥样，如果有注入点，那么单独插入一个单引号，破坏原有结构，肯定会造成语句异常。所以单独一个单引号返回是异常的。但是你插两个单引号的时候，因为这俩单引号闭合了，所以基本不会影响原语句，还是能得到一个相对正常的结果。所以如果发现插入’和’’的返回结果不同，就要多注意了！

load_file()函数

load_file()不仅能读取本地文件也能通过UNC的方式读取远程文件

Load_file函数状态：

当secure_file_priv为空，则表示没有任何限制。

当secure_file_priv为指定目录，则表示数据库导入导出只能在指定目录。

当secure_file_priv为null，则表示不允许导入导出（MySQL 5.7 默认值）

原理

构造语句 调用load_file函数去访问一个攻击者指定的地址 触发dns查询(里面涉及到dns查询的流程 现在本地查询 肯定查询不到 往上一步一步查询就查到了攻击者指定地址的dns服务器上面) 如果这个地址的dns服务器被控制 攻击者就能接受到这个请求 并且得到里面的sql查询语句的结果

本质

本质上是利用了load_file函数能引发外部网络通信的原理 不仅仅是dns协议这个通道 也不仅仅是load_file这一个函数

字节问题

单字节字符集 : 所有的字符都用一个字节来表示 一个字节8比特 也就是8个二进制数

例如 ASCII码

多字节字符集 : 有一部分字符是多个字节来表示 另外一部分是用单字节表示(可能没有 可能全是多字节)

例如 UTF-8 编码 : 使用1~4个字节表示一个符号

GB2312:2字节 UTF-16:2/4字节 GBK

原理

在GBK编码中，一个汉字占用两个字节，其编码范围为首字节0x81-0xFE，尾字节0x40-0xFE（除0x7F）14。当MySQL使用GBK编码时，会将两个连续的字节识别为一个汉字，前提是前一个字节的ASCII码要大于128

在数据库里面这样会导致原本为了转义单引号而添加的反斜杠 与前一个字符合并 导致单引号逃逸出来 原本反斜杠转义单引号的功能不再起作用

本质

这里的GBK编码 指的是数据库连接层或处理逻辑的GBK编码 使用了GBK编码 而不是前端网页编码 也不是数据库内部储存的编码

数据库连接层指的是应用程序与数据库之间建立通信通道时涉及的配置和协议

可以简单的抽象理解成这样:

php java(或者别的能对接mysql服务的东西)为客户端 mysql服务为服务端

当客户端想实现数据库相关操作时 会通过某种方式去连接mysql服务 形成一个sql语句的传输通道 那么这个通道就可以理解为数据库连接层(其实有点像下面的预编译的数据传输)

那么正是这个通道的数据传输的格式才影响了sql语句的解析方式 以及最后是否可以被宽字节注入

通常 这个连接层的编码可以这样被设置

payload

二次注入

本质

第一次通过某些方式(可能是插入数据 也可能是修改数据)将恶意的sql语句插入数据库里面存储 第二次再去查询这部分数据 造成注入

第一次

提交sql语句时直接提交恶意语句 此时语句里面的某些字符 例如单引号之类的 会被addslashes()函数在前面添加反斜杠进行转义

但是这个反斜杠是不会储存到数据库内部的 他的作用只是限制在了提交的那一刻 最终数据库内部还是原始的payload语句(显然这个时候的恶意语句已经没有任何作用了)

第二次

在将恶意payload存到了数据库内部之后 开发者认为在数据库内部的东西一定是可信的 没有危害的

于是从数据库内部取出来的东西 (例如通过sql语句查看某个已经注册了的用户的信息 会用到select去查询数据库内部的信息)是不会进行进一步的检验和处理 也不会进行过滤啥的

然后就导致了二次注入

延时注入

或者说时间盲注

属于盲注的一种 原理是利用时间的长短来判断SQL注入点是否存在的，根据时间的长短来得到SQL语句的执行结果是True还是False

是通过根据数据库执行操作的时间最终体现为页面回显的时间长短(但是本质还是数据库进行操作的时间) 来判断sql语句是否执行 或者 sql语句是否执行成功

目前市面上大多数WAF都是基于规则的检测系统1。WAF的工作流程是：解析HTTP请求 → 匹配规则 → 防御动作 → 记录日志16。

WAF对接收到的数据包进行正则匹配过滤，如果正则匹配到与现有漏洞知识库的攻击代码相同，则认为这是恶意代码，从而进行阻断16。这种检测方式的核心挑战在于识别率——对于已知的攻击方式可以谈识别率，但对于未知的攻击手段，WAF是检测不到的16

分类

非嵌入型 WAF 与 嵌入型 WAF

非嵌入型 WAF

也叫旁路型或代理型 WAF。它作为独立的设备或服务存在，不和 Web 应用混在一起。

云 WAF（如阿里云云盾、Cloudflare）

硬件 WAF（如 F5、绿盟 WAF 网关）

软件型 WAF（如 Nginx + ModSecurity、OpenResty）

特点

与web应用分隔 独立运行

独立完成上述的工作流程

通常起作用在web应用之前 也就是先进行检测 再决定是否放行或者是丢弃或者是拦截

嵌入型WAF

内置 WAF，它是以 SDK、库或代码形式集成在 Web 应用程序内部的

例如

Java 中的防护框架（如 WAF 组件包）

PHP 内嵌防注入类库（如 AntiSQLInject）

某些 CMS 自带的安全检测逻辑

特点

拿到的参数输入是已经被web应用程序做了一定的操作 例如解码、处理、解析 之后的内容了

要想在WAF解析HTTP请求阶段绕过，首先，我们得了解HTTP报文的各个字段的含义和服务器对HTTP报文各字段的解析。

有关于HTTP数据的分块编码：HTTP协议之分段传输与分块编码

提什么权

因为一般来讲 数据库服务都是由高权限用户启动的 以高权限运行 例如

  • Windows系统：MySQL服务以system权限运行
  • Linux系统：MySQL服务以root权限运行

但是即使是获得了数据库的root权限 也只能执行sql命令 能干的事情太少了

因为我们的最终目的是整台服务器的最高权限 而不是这台服务器上的某个服务的最高权限

所以说 提权的目的是利用得到的这个数据库权限，想办法获得服务器操作系统的管理员权限 或者是 执行系统命令

UDF提权

全称为user defined function，用户自定义函数

自定义函数，是数据库功能的一种扩展。用户通过自定义函数可以实现在 MySQL 中无法方便实现的功能，其添加的新函数都可以在 SQL 语句中调用，就像调用本机函数 version () 等方便。

原理

通过自己编写恶意的动态链接库文件(so文件或者是dll文件 里面包含了自定义的恶意的函数 通过这些函数能达到我们执行系统命令的效果) 加载到数据库里面 最终引入这个动态链接库文件 就可以执行命令了

MOF 提权

原理

C:/Windows/system32/wbem/mof/ 目录下的 mof 文件每 隔一段时间（几秒钟左右）都会被系统执行，因为这个 MOF 里面有一部分是 VBS 脚本，所以可以利用这个 VBS 脚本来调用 CMD 来执行系统命令，如果 MySQL 有权限操作 mof 目录的话，就可以来执行任意命令了

预编译

https://www.cnblogs.com/Createsequence/p/16963891.html

本质

本质是 将sql语句和用户的输入进行严格分离

避免用户的输入被当做sql语句的一部分执行

原理

将SQL语句的语法结构与数据参数分离，通过占位符实现参数化查询

先设定sql查询模版语句

然后发送到数据库服务器进行编译 编译完之后等待用户的输入

将用户的输入 通通作为 预先设定好的sql查询语句的参数部分

而不会带入到查询语句当中

过程中 即使用户输入包含恶意语句 如' OR '1'='1 最后也只会被当做字符串处理

关于这个当成字符串处理

传输参数时 并不是进行简单的转义 而是被转成了二进制格式发送给服务器

这样彻底断绝了注入的可能 它们在协议层面就被标识为字符串类型的二进制数据 而不是SQL代码。

java里面的jdbc版本

当然这里只说了java 和 php 上层的事 实际上更重要的是数据库底层如何处理这些预编译的语句 如何准备接受传参 如何解析 如何再去查询

这里就相当于是java php调用了数据库的接口去实现了预编译的效果 实际的处理逻辑还是在数据库底层当中
