docker未授权访问

docker 本质上是一个c/s架构

Docker 客户端（CLI）：运行 docker 命令。

Docker 守护进程（Docker Daemon）：接受命令，实际执行操作（启动容器、拉取镜像等）。

二者之间靠 Docker Remote API 通信 默认通过 Unix Socket 或 TCP 端口 (2375/2376)

平时使用的docker命令都是客户端的命令

而实际上都是向docker的服务端发送请求 服务端收到后 再用dockerd执行命令

Docker Daemon 可配置监听 TCP 端口（2375），方便远程管理

监听在服务器的2375端口上 如果没做任何限制 没有任何鉴权 就能可以被任何人用docker客户端控制这台主机(知道公网ip)

挂载(mount)

是 Linux 系统中将一个“目录”与某个“实际存储位置（磁盘、分区、设备）”关联起来的操作。

简单来说：

  • / 是系统的根目录

意思就是 在容器内访问 /mnt/etc/passwd，你实际是在访问宿主机的 /etc/passwd

docker逃逸

Docker逃逸是指攻击者在获得容器内某种权限的命令执行能力后，通过特定手段突破容器隔离限制，进而获取容器所在宿主机上的权限和资源访问能力的过程。由于Docker使用的是隔离技术，正常情况下容器内的进程无法看到外部的进程，但外部进程可以看到容器内的进程，因此当容器能够访问到外部资源或获得宿主机权限时，就发生了Docker逃逸

容器和虚拟机的区别

逃逸之前先得讲讲这个

虚拟机通过添加Hypervisor层（虚拟化中间层），虚拟出网卡、内存、CPU等虚拟硬件，再在其上建立虚拟机，每个虚拟机都有自己的系统内核。
而Docker容器则是通过隔离的方式，将文件系统、进程、设备、网络等资源进行隔离，再对权限、CPU资源等进行控制，最终让容器之间互不影响，容器无法影响宿主机

虚拟机自己虚拟出自身需要的硬件设施 而容器与宿主机共享内核、文件系统等资源

如何逃逸

目前产生Docker逃逸的原因总共有三种：

  1. 由内核漏洞引起。
  2. 由Docker软件设计引起。
  3. 由特权模式与配置不当引起。

内核引起的逃逸

上述讲到 容器内部的设施是共享宿主机的 所以当宿主机的内核等等存在安全问题 同样也会影响到内部

Dirty Cow（CVE-2016-5195）

由Docker软件设计引起的逃逸

  • CVE-2019-5736（runc）
  • CVE-2020-15257（containerd）

由于特权模式+挂载不当引起的逃逸
